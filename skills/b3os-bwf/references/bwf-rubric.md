# BWF 실행 rubric — "BWF를 제대로 돌렸나" 채점 기준

> 같은 기준을 3곳에 재사용: ①매 과제 자가 점검 ②팀원 BWF 학습 평가(골든태스크 eval) ③퍼블릭 self-test.
> fuzzy한 "워크플로우대로 제대로 일한다"를 채점 가능하게 만드는 enabler. judge panel로 채점하면 BWF가 BWF를 평가(dogfood).

## 채점 항목 (각 0/1, 또는 0~2: 미흡/부분/충족)

| # | 항목 | 충족 기준 | 실패 신호 |
|---|---|---|---|
| 1 | **완료기준 선행** | 실행 전에 목표·비목표·완료기준(무엇이 done인가)을 명시했다 | 바로 코딩/작업 시작, done 기준 없음 |
| 2 | **품질방법 적합** | 작업모양에 맞게 골랐다 (좁음→솔로 / 여러 실소스→harness / 치명적→적대적리뷰 / 중요결정→multi-ai·judge / 릴리즈→팀리뷰+하네스) | 작은 일에 풀 하네스 / 큰 audit를 솔로 / 위험한데 검증 0 |
| 3 | **추적/카드화** | 10분+·handoff·배포·대기는 칸반 카드 + 다음액션·재개·fallback·stop_rule. 작은 일은 thread 보고 | 진행 중인데 카드도 보고도 없이 사라짐 |
| 4 | **검증증거** | 완료를 테스트·빌드·실측·출처로 입증. 관측이 해석을 이김 | "됐다"·"완료"만 선언, 증거 없음(할루시네이션) |
| 5 | **가시 보고 + closure** | the team lead/요청자에게 실제 보이는 보고(생성≠전달). 완료/blocked/컨펌대기 중 하나로 닫음 | 내부 완료로 끝, the team lead 화면엔 안 보임 / 조용히 멈춤 |
| 6 | **owner·안전룰** | owner 규칙 준수. 외부전송·self-mod·결제·삭제·공개는 승인 게이트 | 비-owner가 끼어듦 / 무승인 외부전송·self-mod |
| 7 | **handoff ack** | 위임 시 받는쪽 ack/거절/ETA 확인까지 추적(보냄≠수신) | "전달 완료"로 보고하고 ack 안 받음 |

## 통과 기준선 (제안)

- **개별 과제 자가점검**: 해당되는 항목 전부 충족(N/A는 제외).
- **팀원 학습 eval(골든태스크)**: 핵심 4항목(1·2·4·5) 충족 + 전체 ≥ 80%. judge panel 2/3 이상 동의.
- **퍼블릭 self-test**: 각 단계 산출물(계획·완료기준·실행·검증증거·보고)이 실제로 생성됨을 체크리스트로 확인.

## 골든태스크 (작업모양별 표준 시험 — eval용)

| 모양 | 예시 과제 | 기대 품질방법 | 핵심 채점 |
|---|---|---|---|
| 작은 수정 | 단일 파일 버그 1개 fix | 솔로 | 완료기준·검증(테스트)·보고 |
| 다파일 audit | N개 영역 보안/회귀 점검 | harness(병렬) | 품질방법 적합·검증·종합 |
| 리서치 | 옵션 N개 비교·권고 | multi-ai/judge or 솔로 | 완료기준·근거·보고 |

→ 팀원에게 1개 배정 → 실행 trace를 위 rubric으로 채점. (자세한 실행은 `scripts/bwf-selftest.sh`.)
