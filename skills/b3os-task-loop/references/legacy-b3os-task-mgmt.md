# b3rys Task 관리 절차 스킬

TEAM-OS 10장의 기본 테스크 룰(task operations, 과제 운영)은 팀원이 별도로 요청받지 않아도 항상 적용하는 기본 원칙이다. 이 스킬은 그 원칙 자체가 아니라, Tasks 칸반(대시보드 `<dashboard-url>/team` -> Tasks, 정본 = team-collab `task` DB), continuation guard(진행 지속 장치), handoff(인수인계) 검증에 적용하는 절차 문서다.

사용자가 "task ops로 해줘"라고 말해야 작동하는 구조가 아니다. 실행 과제·위임·피드백 수집·보고 정리는 TEAM-OS 기본 룰로 자동 적용하고, 이 스킬은 카드 작성법이나 guard가 읽을 필드처럼 구체 절차가 필요할 때 참조한다.

## 언제 쓰나

- the team lead가 컨펌한 실행 과제를 칸반에 등록하거나 갱신할 때
- 진행 중인 작업, 현재 상황, 열린 과제, 누가 하던 일을 요약할 때
- `다음 액션`, `재개 시각`, `fallback`, `stop_rule`처럼 guard가 읽는 필드를 남길 때
- 과제가 blocked(막힘) 상태인지 description(설명)에 표시할 때
- 다른 팀원에게 작업·검증·배포·조사를 handoff할 때
- 같은 요청/보고가 중복으로 나갈 위험이 있어 기존 thread(대화 묶음)와 message DB(메시지 데이터베이스)를 확인할 때

**챗방 논의·구체화 단계는 칸반에 입력하지 않는다.** the team lead 컨펌 = 실행 시작 = 그때 오너가 입력한다.

작업 카드는 모든 작은 일에 강제하지 않는다. 10분 이상 걸리거나, handoff(인수인계), 배포, 실환경 확인, 대기/재개가 생기면 카드로 남긴다. 그보다 작은 일은 같은 thread(대화 묶음)에 owner(담당자), 다음 액션, 완료 근거만 남겨도 된다.

상태 요약은 예외 없이 칸반을 먼저 본다. the team lead가 "진행중인 작업", "현재 상황", "열린 과제", "누가 하던 일"을 물으면 기억이나 최근 대화만으로 답하지 말고 `/team/api/tasks` 또는 `team.db task`를 조회한다. 답변은 `doing / plan / 최근 중요 done / 보류·대기 / 카드 밖 즉시완료 예외`로 재구성한다. 발표자료/덱, Team Search, learning-loop처럼 큰 작업 축은 카드가 있으면 반드시 포함한다.

## 기본 원칙

- Owner(담당자)는 1명이다. 공동작업은 description에 적고, 최종 추적 책임자는 한 명으로 둔다.
- 일을 받으면 완료 기준을 먼저 맞춘다. 코드 완료, 배포 완료, 실환경 확인은 서로 다른 gate(완료 단계)이므로 어느 단계까지가 이번 완료인지 적는다.
- 받을 수 없는 일은 조용히 두지 않는다. 거절·재배정 사유와 추천 owner를 남긴다.
- 실행단계 기본값은 `주행모드`다. 한 번 답하고 끝내지 않고 완료/blocked(막힘)/컨펌 대기까지 들고 간다.
- 주행모드는 새 일을 만드는 모드가 아니다. 끝나지 않은 일이 조용히 사라지는 것만 막는다.
- 커뮤니케이션 owner 판단이 먼저다. 내가 owner가 아니면 PM 관점이라도 끼어들지 않는다.
- 칸반은 현황판이다. 세부 PM 툴처럼 필드를 늘리지 않고 description에 필요한 맥락만 둔다.

## 작업 카드 4필드 (이것만)

| 필드 | 내용 |
|---|---|
| **제목(title)** | 결과물/행동 기준 한 줄. 예: "라우터 reply-owner 구현" |
| **담당자(owner)** | 최종 책임자 1명. 공동작업이면 description에 적고 owner는 1명. |
| **상태(column)** | 계획 / 실행 중 / 완료. (blocked는 별도 컬럼 없이 description 첫 줄 또는 배지로) |
| **description** | 아래 미니 템플릿(자유형식). 카드 펼쳐서 작성·수정. |

서브과제·실행계획을 별도 필드로 쪼개지 않는다(= PM 툴화 방지). description 안에 둔다. 단, 실행 중 과제는 조용히 멈추지 않도록 `다음 액션` 블록을 남긴다.

작은 일은 풀 템플릿을 강제하지 않는다. 한 번의 명확한 행동으로 끝나는 일은 `제목 / owner / 상태 / 다음 액션`만 남긴다. 여러 사람 handoff, 배포, 검증, 재개가 필요한 일은 아래 풀 템플릿을 쓴다.

## description 미니 템플릿

```
목표: 무엇을 끝내면 되는지 (완료 시 결과물)
범위: 이번 카드에 포함/제외할 것
계획:
- 2~5개 짧은 단계
완료 기준: done으로 옮기는 조건. 코드/배포/실환경 중 어느 gate까지인지 명시
다음 액션: 지금 이어서 할 한 가지 행동. 없으면 컨펌 요청/대기 사유를 적는다.
재개 시각: 기다리는 일이 있으면 다시 볼 시각. 예: 2026-06-02 10:30 KST
fallback: 재개 시각에도 안 풀리면 할 대체 행동
stop_rule: 같은 실패/무응답이 몇 번이면 멈추고 요약 보고할지
검증 증거: 완료 판단 근거 1줄. 예: 테스트명, 산출물 경로, 배포 URL, 실기기 확인. 작업 카드가 있으면 이 줄이 정본
메모: 링크·결정·주의점 (blocked면 첫 줄에 "🚫 blocked: 이유")
```

자동 guard(진행 지속 알림)가 읽는 필드는 줄 시작을 유지한다. 예: `재개 시각: 2026-06-02 10:30 KST`처럼 쓴다. 괄호 설명이나 긴 메모는 `메모:`에 둔다.

## 상태 기준

- **계획**: the team lead 컨펌됐으나 아직 미착수. (논의 중인 건 칸반에 없음)
- **실행 중**: 오너가 작업 시작. `다음 액션`은 항상 있어야 한다. blocked면 description에 "🚫 blocked: 이유"를 첫 줄에 두고, `재개 시각`/`fallback`도 같이 둔다(컬럼은 실행 중 유지).
- **완료**: description의 "완료 기준"을 충족하고 `검증 증거`가 남았을 때만. 코드 완료, 배포 완료, 실환경 확인은 서로 다르므로 어느 gate(단계)를 통과했는지 적는다.
- **드랍(취소)**: 카드를 삭제. (감사가 필요하면 description에 "dropped: 이유" 남기고 완료로 옮긴 뒤 삭제 보류는 the team lead 판단)

## 주행모드와 continuation guard

자율주행은 3단계로 본다.

- `턴모드`: 한 번 묻고 한 번 답하는 대화·검토 단계.
- `주행모드`: 실행 단계의 기본값. owner 또는 coordinator PM이 다음 액션·재개 시각·fallback·stop_rule을 들고 간다.
- `완전자율모드`: 기획부터 실행, 검증, 보고까지 넓게 위임받는 상위 모드. 범위·권한·stop_rule을 더 명확히 둔다.

주행모드의 핵심 문제는 "컨펌을 요청하지도, 알아서 진행하지도 않고 조용히 멈추는 상태"다. 그래서 실행 중 과제는 종료/보고 전 항상 아래를 확인한다.

- 남은 작업이 있으면 `다음 액션`을 적고 진행한다.
- 컨펌이 필요하면 `다음 액션`에 누구에게 무엇을 물을지 적는다.
- 기다려야 하면 `재개 시각`과 `fallback`을 적는다.
- 같은 실패/무응답 반복은 `stop_rule`에 따라 요약 보고하고 멈춘다.

`재개 시각`이 지난 과제나 `다음 액션`이 없는 실행 중 과제는 continuation guard 스크립트가 owner에게 버스 DM으로 알려준다. 이것은 새 PM 시스템이 아니라 칸반의 description을 읽는 얇은 노티 장치다.

매일 06:00 KST의 `task-review-ping`은 owner별 active 카드를 다시 보여 주는 일일 백스톱이다. owner는 완료된 doing을 done으로 옮기고, 남은 doing의 다음 액션을 갱신하며, 오래된 plan은 실제로 하지 않을 때만 폐기한다. 정리할 것이 없으면 억지 수정이나 보고를 하지 않는다.

Guard ping(진행 지속 알림)은 안 끝난 일이 사라지는 것을 막기 위한 것이다. 이미 완료됐거나, 변화가 없고 같은 미해결 상태가 active(진행 중)로 잡혀 있으면 반복 ping을 만들지 않는다.

## Handoff 검증

다른 팀원에게 맡겼다고 owner 책임이 끝나지 않는다. handoff는 아래 중 하나가 기록되어야 완료다.

- 받은 팀원이 접수했고, 다음 액션 또는 ETA(예상 완료 시각)를 남김.
- 받은 팀원이 결과와 검증 근거를 돌려줌.
- 기다리는 상태라면 `재개 시각`, `fallback`, `stop_rule`을 남김.
- 막혔다면 blocked 이유와 다음 판단자가 누구인지 남김.

Owner 전환은 받은 쪽의 ack(인수 확인) 전에는 끝나지 않는다. 새 owner가 "접수/다음 액션/ETA" 중 최소 하나를 남기기 전까지 이전 owner가 추적 책임을 유지한다.

Handoff 메시지는 짧고 실행 가능해야 한다.

- 팀원에게 보내는 directed message(지정 메시지): 작업 지시, 완료 기준, 필요한 근거만.
- 팀방 visible reply(보이는 답변): the team lead에게 필요한 상태 요약만.
- handoff 카드 또는 메시지에는 현재상태, 다음 액션, 검증된 것/아직 안 된 것, 관련 파일/링크, 다음 owner를 남긴다.
- 같은 긴 본문을 bus DM과 팀방에 복붙하지 않는다.
- 이미 팀방에 보고할 예정이면 bus DM에는 "접수/다음 액션" 중심으로 남긴다.
- 같은 과제에서 새 지시를 보낼 때는 기존 thread/in_reply_to를 이어가고, 새 DM을 여러 개 만들지 않는다.

## 피드백 요청

스킬·운영 원칙·UI·문서에 대한 피드백은 기본적으로 "실제 사용자 입장"으로 요청한다. 팀원의 역할로 관점을 먼저 제한하면 좋은 피드백이 좁아진다.

- 기본 질문: "실제로 쓴다고 생각하고 막히는 점, 빠진 점, 헷갈리는 점, 중복될 것 같은 점을 자유롭게 봐 주세요."
- 역할 기반 관점은 보조 질문으로만 붙인다. 예: "추가로 인프라 관점에서 특별히 위험한 게 있으면 알려 주세요."
- 피드백 요청을 여러 팀원에게 보낼 때도 각자에게 다른 역할 박스를 씌우지 않는다.
- 피드백 종합은 개수 제한이 아니라 핵심성으로 한다. 길어진 합의 내용을 그대로 나열하지 말고, 실제 판단·실행에 필요한 핵심만 다시 뽑는다.
- 받은 피드백은 개인 의견으로만 두지 않고, 반복되는 교훈이면 이 스킬 또는 `rules/SHARED.md`로 승격한다.

## 중복 방지

주행모드에서 가장 흔한 실패는 같은 owner가 여러 경로로 같은 팀원을 깨우는 것이다. 새 메시지를 보내기 전 아래를 확인한다.

- 이미 같은 과제·같은 수신자에게 보낸 handoff가 있는가?
- 기존 thread에 reply로 이어갈 수 있는가?
- cron(예약 확인), 수동 확인, OpenClaw injection(주입) 재시도가 동시에 돌고 있지는 않은가?
- 팀방에 보일 요약과 팀원에게 보낼 작업 지시가 분리되어 있는가?
- 동일한 본문을 다시 보내야 한다면 dedupe_key(중복 방지 키) 또는 명시적 in_reply_to를 둔다.
- 실패처럼 보여도 재전송 전에 최근 `message` DB(데이터베이스)와 audit log(감사 로그)를 먼저 조회한다. 다른 coordinator/OpenClaw 경로가 이미 보냈을 수 있다.

중복이 의심되면 먼저 새 지시를 멈추고, audit log(감사 로그)와 기존 thread를 확인한 뒤 하나의 경로만 남긴다.

## 흐름 (TEAM-OS 10장)

1. 챗방에서 논의·구체화 (입력 X)
2. 오너가 위 포맷으로 정리
3. **the team lead 컨펌 = 실행 시작**
4. 오너가 칸반에 입력 (대시보드 또는 API)
5. 이후 **오너가 책임지고 갱신** (매번 묻지 않게)

## 어떻게 (입력·갱신)

- **대시보드**: /team → Tasks. 컬럼별 "+ 과제 추가"(제목+담당자), 카드 ▸ 펼쳐 description 작성, 담당자 드롭다운 변경, ◀▶ 컬럼 이동, ✕ 삭제. 담당자 필터로 내 과제만 보기.
- **API**(자동화/스크립트): `POST /team/api/tasks {title, column, owner, description}` · `PATCH /team/api/tasks/:id {title?, column?, owner?, description?}` · `DELETE /team/api/tasks/:id`. column = plan|doing|done.

## 예시

**제목**: team-collab 검색(FTS) V1 구현
**담당자**: agent-b
**상태**: 실행 중
**description**:
```
목표: 팀 버스 메시지를 한글 토크나이저로 검색 가능하게
범위: 포함=bus messages FTS · 제외=개인 MEMORY
계획:
- FTS5 가상테이블 마이그레이션(additive)
- 한글 토크나이저·청킹
- 검색 API + 대시보드 연결
완료 기준: 대시보드에서 한글 키워드 검색 → 관련 메시지 반환, reindex가 dispatcher 안 막음
다음 액션: FTS5 마이그레이션 초안 작성 후 로컬 테스트
재개 시각:
fallback: 로컬 테스트가 막히면 원인 로그와 대안 2개를 maintainer에게 리뷰 요청
stop_rule: 같은 마이그레이션 실패 3회면 중단하고 the team lead/maintainer에게 요약 보고
메모: coordinator=PM, reindex는 throttled one-shot(write-lock 주의)
```

## 주의

- 칸반을 PM 툴로 키우지 않는다. 필드를 늘리거나 서브과제를 구조화하지 않는다 — 늘릴 게 있으면 description으로.
- 내 과제가 아니면 owner의 카드를 임의로 바꾸지 않는다(상태/내용은 owner 책임).
- 다른 팀원에게 맡긴 작업도 내 PM 책임이면 handoff 검증까지 보고 끝낸다.
- 팀원 경험에서 반복되는 교훈은 개인 메모리에만 두지 말고 이 스킬 또는 `rules/SHARED.md`에 반영한다.
