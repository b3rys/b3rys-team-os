WORKFLOW

일이 조용히 사라지지 않게

BWF · Kanban · Autonomy · Handoff · Harness

b3rys 팀이 과제를 받아 끝까지 닫는 실행 체계. 6단계 BWF, 3컬럼 칸반, 자율주행 3단계, 핸드오프 회수, Harness 병렬 실행, Proposal 거버넌스까지 continuation guard(진행 지속 장치)로 연결한다.

01 — BWF

BWF 6단계 흐름

BWF(b3os workflow, 팀 기본 과제 수행 흐름) — GD가 컨펌한 실행·위임 과제를 끝까지 닫는 6단계 사이클.

01
PM 계획·착수
착수 즉시 칸반 카드 등록. GD 리마인드 없이 자동.
02
팀배정·handoff
적임자 배정 + handoff ack(접수 확인) 추적.
03
실행·품질방법
harness / multi-ai / judge / 적대적 리뷰 중 선택.
04
검증
실제 검증 · 다관점 리뷰 · 빈틈없는 테스트.
05
보고·closure
사용자 가시 결과 기준으로 종료(closure).
06
학습 hook
교훈을 SHARED.md로 흘려보내는 후크.
Continuation Guard
PM은 착수 · handoff · reviewer ack · gate 전이마다 GD-visible 1줄 보고로 진행 지속 장치를 세운다. 끝나지 않은 일이 조용히 사라지지 않게.
BWF 예외
단순 답변 · 조회 · 즉시 끝나는 일은 BWF를 돌리지 않는다. 6단계는 컨펌된 실행·위임 과제에만 적용.
02 — KANBAN

칸반 시스템

Tasks 칸반(`task` DB 정본)은 PM 툴이 아니라 현황판. 3컬럼 plan / doing / done.

plan
착수 대기 · 보류 · 컨펌 대기 카드. 논의·구체화 단계는 칸반에 넣지 않는다.
doing
진행 중. owner가 직접 상태를 갱신. blocked 는 별도 컬럼 없이 배지로 표시.
done
사용자 가시 결과까지 도달한 카드. 최근 중요 done 은 PM 요약의 한 축.
CARD ANATOMY
flat owner task
제목 / 담당자 1명 / 상태 / description. 공동작업은 description 에 적고, blocked 는 배지로. owner 가 자기 상태를 직접 갱신.
"제목 · owner 1 · doing · desc" — 담당자는 항상 1명, 공동은 본문에
PM 상태 요약
칸반 먼저 조회
GD가 진행 상황을 물으면 칸반을 먼저 조회 → doing / plan / 최근 중요 done / 보류·대기로 재구성. 칸반에 없다고 없는 일로 단정 금지.
"누락 발견 = 즉시 카드화" — 06:00 KST task-review-ping 은 사후 점검
03 — AUTONOMY

자율주행 3단계

실행 단계 기본값은 주행모드. harness 확장 개념과 1:1로 연결된다.

턴모드 harness off
한 번 묻고 한 번 답하는 대화·검토 단계. 솔로 실행, 확장 없음.
주행모드 기본값 · limited
다음액션 · 재개시각 · fallback · stop_rule 명시. 완료 / blocked / 컨펌대기까지 owner가 들고 간다. limited harness.
완전자율모드 full
기획~실행·검증·보고까지 넓게 위임. 범위·권한·stop_rule 을 더 명확히. full harness.
목적
일을 계속 만드는 게 아니라, 끝나지 않은 일이 조용히 사라지는 걸 막기.
모드 ↔ harness
턴 → off · 주행 → limited · 완전자율 → full. 넓거나 교차검증 필요하면 harness, 좁고 순차면 솔로.
04 — HANDOFF

핸드오프 & Continuation Guard

handoff(인수인계)는 보낸 것으로 끝나지 않는다. 성공 기준은 '전달 완료'가 아니라 회수 완료.

01
봉투 발송
to · source_thread · context · task · done_when · fallback.
02
receiver ACK
4선택 중 1회만. ACK 전까지 owner=sender.
03
실행
ACCEPT 후에도 sender가 closure 책임 유지.
04
완료보고
result · evidence · remaining_risk · reported_to.
05
closure
complete / blocked / confirm_wait 중 하나로 닫기.
HANDOFF ENVELOPE
인계 봉투
to / source_thread / user_visible_context / task / done_when / constraints / reply_by / fallback / visibility_target.
"누구·맥락·할일·완료기준·기한" — 9개 필드를 한 봉투로
COMPLETION REPORT
완료 보고
result / evidence / remaining_risk / reported_to. 생성됨과 GD에게 보임은 구분한다.
"성공 = 회수 완료" — 전달 완료가 아니라 닫힘까지
ACK ①
ACCEPT + 다음액션 + ETA
ACK ②
DECLINE + 사유 + 추천 owner
ACK ③
NEED_INFO + 질문
ACK ④
FIRST_TAKE (1차 의견)
Continuation Guard
reply_by 내 ack 없으면 sender 재알림. ETA 초과면 receiver ping. complete / blocked / confirm_wait 로 닫을 때까지 추적.
ACK는 1회만
도배 금지. 수집형 피드백은 조용히 모은 뒤 종합 1회 보고로 루프를 닫는다.
05 — WORKLOOP

작업루프 — 반복 운영

정해진 시각에 담당자를 깨우고, 한 턴 안에서 루프를 닫는 반복 운영 사이클. 스킬: b3os-workloop

01
wake
운영 시스템이 정해진 시각에 담당자를 깨움.
02
상태 조회·검증
실제 상태 확인. 자동화 핑은 복구·완료가 아니다.
03
한 턴 닫기
완료 / 갱신 / 보고 / blocked / 컨펌대기 / 다음 wake 예약.
담당 결정
capability / role 기준. 불명확하면 coordinator fallback, 없으면 가용 팀원(축소 운영임을 GD에 명시).
무기한 금지
stop_rule / 만료 없이 무기한 지속하지 않는다. 실패가 반복되면 에스컬레이션.
06 — HARNESS

Harness — 병렬 실행

한 팀원이 sub agent(보조 에이전트)를 병렬로 띄워 분리 가능한 일을 나눠 처리하는 실행 방식. 스킬: b3rys-harness

MODE MAP
모드 ↔ harness
턴 → off · 주행 → limited(병렬 실소스 커버가 기본) · 완전자율 → full.
"넓거나 교차검증 → harness" — 좁고 순차면 솔로
SPEC FIELDS
카드·지시 명시
harness · subagents · budget · scope · verify 를 카드/지시에 명시한다.
"큰 fan-out · full = GD 고지 + stop_rule" — 실행 전 필수
off 턴
솔로 실행. 좁고 순차 작업.
limited 주행
병렬 실소스 커버 모양이면 기본. 분리 가능한 일만 fan-out.
full 완전자율
대량 병렬. 실행 전 GD 고지 + stop_rule 필수.

gd_report

05
accepted / rejected

REVIEW PATH

08 — SELF-LEARNING

Self-Learning 주간 루프

교훈을 팀 정책으로 승격시키는 금요일 루프. 메인 목적은 팀 정책·피드백 루프의 자가발전.

STEP 1 — 발견
교훈 발견 → SHARED.md append
작업 중 나온 교훈을 append-only(추가 전용)로 기록. 일회성 잡음은 제외.
STEP 2 — 큐레이션
큐레이터 검토
반복·고정 교훈만 TEAM-OS 승격 후보로 추린다. 별도 후보 장부 없이 항목 상태로 관리.
FRI 05:00 KST
후보 수집·정리·공동리뷰 요청
금요일 05시에 후보를 모으고 공동 리뷰를 요청. 지연되면 지연 상태와 임시 판단을 분리 보고.
FRI 10:00 KST
learning-loop PM → GD 보고
learning_loop_pm 보유자(없으면 coordinator fallback)가 정리된 결과를 GD에 보고.
STEP 5 — 반영
GD 승인 → TEAM-OS 반영
정책·보안·라우터·외부전송 관련은 GD 승인 없이 반영 금지. 월 1회 작업성과 개선지표 확인.
09 — GLOSSARY

용어집

이 페이지에서 쓰인 핵심 용어 정리.

BWF
b3os workflow. 컨펌된 실행·위임 과제를 끝까지 닫는 팀 기본 6단계 흐름.
Harness
한 팀원이 sub agent를 병렬로 띄워 분리 가능한 일을 나눠 처리하는 실행 방식.
Handoff
인수인계. ack·거절·ETA 확인 전까지 owner가 넘어가지 않는다.
Proposal
팀 제안. draft→peer→pm→gd_report→결정의 상태기계로 관리.
Peer Review
동료 리뷰. 반대리뷰(is_adversarial) 1건 이상이 pm 게이트 통과 조건.
PM Review
PM 검토. Bill 우선, 제안자가 Bill이면 Codex가 받음.
GD Report
Codex PM만 수행하는 GD 보고 단계. 결정은 GD(actor=gd)만.
Revise Requested
수정 요청 상태. draft로 되돌아간다.
Self-Learning
교훈→SHARED.md→큐레이션→TEAM-OS 승격의 주간 자가학습 루프.
Continuation Guard
진행 지속 장치. complete/blocked/confirm_wait로 닫을 때까지 추적.
Limited Harness
주행모드 기본. 병렬 실소스 커버 모양에서 제한적 fan-out.
Full Harness
완전자율모드. 대량 병렬, 실행 전 GD 고지 + stop_rule 필수.
