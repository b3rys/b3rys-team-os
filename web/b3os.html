Self-Hosted AI Team OS

b3os — 팀이 일하는 방식을 호스팅한다

Kraken gives agents a room. b3os gives them an operating doctrine.

'AI 에이전트 채팅툴'이 아니라 self-hosted AI team operating system(직접 호스팅하는 AI 팀 운영체계). 유료 가치는 모델 성능이 아니라 신뢰·운영·감사·셀프서비스 영입·업데이트/지원. 본체는 대시보드 기반 운영(영입·OT·설정·과제·보고)이고, 텔레그램·슬랙은 대화 창구일 뿐이다.

01 — VISION & POSITIONING

운영 도크트린을 파는 제품

경쟁(크라켄)은 협업 '방'만 준다. b3os는 팀이 일하는 규칙과 운영체계 — 운영 도크트린(operating doctrine, 운영 교범) — 을 함께 제공한다.

제품 본체 = 대시보드 운영
영입 · OT(온보딩) · 설정 · 과제 · 보고를 대시보드에서 처리. 채널(텔레그램/슬랙)은 입출력 창구이지 제품이 아니다.
유료 가치 = 운영 레이어
신뢰 · 운영 · 감사 · 셀프서비스 영입 · 업데이트/지원. 모델 성능은 과금 대상이 아니다. LLM API·호스팅은 고객 부담.
Kraken — 방만 준다
협업할 공간(room)만 제공. 팀이 어떻게 일할지의 규칙·운영체계는 사용자가 알아서.
b3os — 도크트린을 준다
owner 규칙·과제 라이프사이클·신뢰 버스·감사·영입 절차까지 운영체계로 패키징.

BEFORE — launchd plist

AFTER — in-process

07 — TEAM MESSAGE BUS

봇 ↔ 봇 통신의 신뢰 레이어

메시지 insert → 수신자 자동 wake. delivery(실전달)와 visibility(표시)를 분리하고, anti-pingpong 가드로 무한 루프를 막는다.

01
pending 폴링
ack_at IS NULL 인 메시지 조회.
02
runtime 재배치
수신자 런타임으로 라우팅.
03
wake adapter
claude=tmuxInject · openclaw=push-wake.
04
retry / backoff
실패 시 1s / 2s / 4s, 최대 3회.
DELIVERY vs VISIBILITY
실전달과 표시의 분리
버스에 배달되는 것과 채널에 보이는 것은 다른 축. telegram sync 정책 = none / status / handoff / result.
ANTI-PINGPONG
무한 왕복 차단
hop_count + parent_message_id 추적. BUS_MAX_AUTO_ROUNDS(기본 2) 초과 시 차단 + owner → GD 알림.

4 안전장치 (safety grid)

원자적 dispatch 마킹
delivery_state = dispatching 으로 중복 발사 방지.
wake timeout + async 격리
~10s 타임아웃, 비동기로 격리.
dead-letter
3회 초과 → failed + audit 기록.
신뢰 출처만 wake
위조 발신자는 깨우지 않음.
08 — NATIVE RUNTIME

b3os가 LLM을 직접 호출한다

외부 런타임 없이 팀원을 구동. 구독 OAuth는 3rd-party 불가이므로 합법 경로만 쓴다.

접근 A — 기본
API 키 (종량 · 깨끗)
공식 API 키로 종량 호출. 약관상 깨끗한 합법 경로.
접근 B — 회색
공식 CLI 래핑 (구독)
구독 기반 공식 CLI 래핑. 회색 지대.
매니지드 비즈모델
고객은 정액 지불 → 우리가 API 키로 종량 호출 → 마진 포함 정액. 비개발자가 키 관리 없이 팀을 굴리게 한다.
